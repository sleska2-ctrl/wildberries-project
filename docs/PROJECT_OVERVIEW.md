# Документация проекта Wildberries Analytics

## Назначение

Проект собирает данные кабинетов Wildberries и Ozon, сохраняет их в локальные SQLite-базы и строит аналитические таблицы для управленческих отчетов. Основной сценарий запускается из CLI или из локального веб-интерфейса.

Текущая основная цепочка:

1. Получить справочник SKU и себестоимости.
2. Забрать данные Wildberries за выбранный период.
3. Нормализовать ответы API в табличный формат.
4. Сохранить сырые таблицы в SQLite без дублей по ключам.
5. Пересчитать производные финансовые и операционные таблицы.
6. Показать результаты в веб-интерфейсе из SQLite.
7. Дать операционные инструменты: планирование, биддер рекламы, задачник и контроль цен.

Внешние таблицы не используются. Актуальная архитектура опирается на SQLite и web UI. Для мультикабинетного режима используются отдельные базы `data/cabs/{cabinet_id}.db` и общая платформа кабинетов.

## Стек

- Python 3.13.
- `requests` для HTTP-запросов к API Wildberries.
- `sqlite3` из стандартной библиотеки для локального хранилища.
- `python-dotenv` для загрузки `.env`.
- `openpyxl` для ручной загрузки Excel-выгрузки воронки WB и экспорта планирования.
- `http.server.ThreadingHTTPServer` для встроенного веб-интерфейса.
- Server-Sent Events в веб-интерфейсе для потоковой передачи логов синхронизации.
- Docker и Docker Compose для серверного запуска веб-интерфейса.

## Конфигурация

Настройки читаются из `.env`.

Обязательные переменные:

- `WB_FINANCE_TOKEN` или общий `WB_API_TOKEN` - токен для finance/statistics/analytics API.
- `WB_ADV_TOKEN` или общий `WB_API_TOKEN` - токен для advertising API.
- `DEFAULT_DATE_FROM` - дата начала периода по умолчанию.
- `DEFAULT_DATE_TO` - дата окончания периода по умолчанию.

Переменные локального хранилища и web UI:

- `SQLITE_DB_PATH` - путь к SQLite, по умолчанию `data/wb_sync.db`.
- `SKU_TABLE` - таблица справочника SKU, по умолчанию `SKU`.
- `RAW_SALES_TABLE` - таблица продаж, по умолчанию `raw_sales`.
- `RAW_ORDERS_TABLE` - таблица заказов, по умолчанию `raw_orders`.
- `RAW_ADS_TABLE` - таблица рекламы, по умолчанию `raw_ads`.
- `DAILY_PNL_TABLE` - таблица PnL, по умолчанию `daily_pnl`.
- `FUNNEL_ANALYTICS_TABLE` - таблица воронки, по умолчанию `funnel_analytics`.
- `WB_SITE_PASSWORD` - пароль web UI, если задан.
- `WEB_PUBLIC_PORT` - внешний порт Docker Compose, по умолчанию `80`.

Переменные и токены мультикабинетной платформы хранятся в кабинете. Для WB-биддера нужен рекламный токен кабинета (`WB_ADV_TOKEN` или кабинетный аналог), потому что исполнитель вызывает `adv/v0/start`, `adv/v0/pause`, `adv/v1/promotion/count`, `api/advert/v2/adverts` и `adv/v3/fullstats`.

Фильтрация артикулов:

- `ARTICLE_FILTER_TYPE` - тип артикула для расчетов, `nmId` или `vendorCode`, по умолчанию `nmId`.
- `ARTICLE_FILTER_VALUES` - CSV-список артикулов. Если пустой, значения берутся из листа/таблицы `SKU`.

Важно: в текущем основном сценарии продажи и заказы сохраняются по полному кабинету. Фильтры и SKU используются прежде всего для маппинга, себестоимости, признака "наши/не наши" и построения отчетов.

## Источники данных

### Wildberries Finance API

Метод в коде: `WildberriesClient.fetch_sales_details`.

Источник: `https://finance-api.wildberries.ru/api/finance/v1/sales-reports/detailed`.

Что собирается:

- финансовый отчет продаж;
- продажи и возвраты;
- суммы продаж;
- комиссии, эквайринг, логистика;
- хранение, приемка, штрафы, удержания, доплаты;
- сумма к перечислению `forPay`;
- связь продажи с датой заказа через `orderDt`.

Данные сохраняются в таблицу `raw_sales`. Ключ upsert: `rrdId`.

### Wildberries Statistics API: заказы

Метод в коде: `WildberriesClient.fetch_orders`.

Источник: `https://statistics-api.wildberries.ru/api/v1/supplier/orders`.

Что собирается:

- поток заказов;
- дата заказа и дата изменения;
- склад, регион, страна;
- `supplierArticle`, `nmId`, barcode;
- цена, скидка, отмена, идентификаторы заказа.

Данные сохраняются в таблицу `raw_orders`. Ключ upsert: `srid`.

### Wildberries Statistics API: остатки

Метод в коде: `WildberriesClient.fetch_stocks`.

Источник: `https://statistics-api.wildberries.ru/api/v1/supplier/stocks`.

Что собирается:

- текущие остатки по складам;
- количество на складе и в пути;
- `supplierArticle`, `nmId`, barcode;
- категория, предмет, бренд, размер;
- цена и скидка.

Данные сохраняются в таблицу `raw_stocks`. Таблица каждый раз заменяется полностью.

### Wildberries Advertising API

Методы в коде:

- `fetch_campaign_ids`;
- `fetch_campaign_details`;
- `fetch_relevant_campaign_ids`;
- `fetch_campaign_stats`.

Источники:

- `https://advert-api.wildberries.ru/adv/v1/promotion/count`;
- `https://advert-api.wildberries.ru/api/advert/v2/adverts`;
- `https://advert-api.wildberries.ru/adv/v3/fullstats`.

Что собирается:

- активные, приостановленные и завершенные рекламные кампании со статусами `7`, `9`, `11`;
- состав кампаний и связанные `nmId`;
- дневная статистика рекламы по товарам: показы, клики, расходы, заказы, сумма заказов.

Данные разворачиваются функцией `flatten_ads_rows` и сохраняются в `raw_ads`. Ключ upsert: `date`, `advertId`, `appType`, `nmId`.

### Wildberries Analytics API: воронка

Метод в коде: `WildberriesClient.fetch_funnel_history`.

Источник: `https://seller-analytics-api.wildberries.ru/api/analytics/v3/sales-funnel/products/history`.

Что собирается:

- история воронки по `nmId`;
- сумма и количество заказов;
- количество и сумма выкупов;
- отмены;
- данные по дням.

Данные сохраняются в `funnel_analytics`. Ключ upsert: `date`, `nmId`.

### Справочник SKU и себестоимости

Источники:

- локальная таблица SQLite `SKU`/`sku`.

Использование:

- ручные атрибуты товара по `nmId`/артикулу поставщика;
- себестоимость для PnL;
- признак `ИИТех` для деления товаров на "наши" и "не наши";
- обогащение SKU полями `Предмет`, `Артикул поставщика`, `Название`, `себестоимость`, `% комиссии на вб`, `Стратегия`, `Категория по выручке`, `склад`.

Поддерживаемые заголовки для WB-артикула:

- `Артикул WB`;
- `nmId`;
- `nm_id`.

Поддерживаемые заголовки для артикула поставщика:

- `Артикул поставщика`;
- `supplierArticle`;
- `SKU`;
- `sku`.

Поддерживаемые заголовки себестоимости:

- `cogs`;
- `себестоимость`;
- `cost_price`.

## Типовая последовательность полной синхронизации

Запуск:

```bash
PYTHONPATH=src python -m wb_gsheets.main --date-from 2026-05-01 --date-to 2026-05-07
```

Последовательность внутри `main.py`:

1. Разбираются CLI-аргументы: даты, `--only`, `--skip-ads`, `--skip-funnel`.
2. Загружается конфигурация из `.env`.
3. Создаются клиенты `WildberriesClient` и `SQLiteStore`.
4. Справочник SKU читается из SQLite.
5. Если в SQLite нет SKU, синк продолжает работу с предупреждением, но справочник извне не подтягивает.
6. Строится маппинг `nmId -> article`.
7. Загружается финансовый отчет продаж за период.
8. Загружаются заказы с `date_from`.
9. Загружаются текущие остатки.
10. Если реклама не отключена, собирается список `nmId`, ищутся связанные рекламные кампании и грузится статистика рекламы.
11. Считается `daily_pnl`.
12. Считаются аналитические таблицы из `analytics.py`.
13. Сырые таблицы `raw_sales`, `raw_orders`, `raw_ads`, `raw_stocks` сохраняются в SQLite.
14. `daily_pnl` сохраняется в SQLite.
15. Если воронка не отключена, загружается `funnel_analytics`.
16. Воронка из API дополняется уже сохраненными строками из SQLite, чтобы старые даты не пропадали из расчетов.
17. Пересобирается `buyout_order_day`.
18. Обновляются производные таблицы:
    - `finance_article_day_detail`;
    - `analytics_article_day`;
    - `analytics_day`;
    - `analytics_article_period`.

## Частичная синхронизация

Параметр `--only` позволяет загрузить только один тип данных, а затем пересчитать производные таблицы из SQLite.

Доступные режимы:

- `--only sales` - загрузить только продажи в `raw_sales`.
- `--only orders` - загрузить только заказы в `raw_orders`.
- `--only stocks` - загрузить только остатки в `raw_stocks`.
- `--only ads` - загрузить только рекламу в `raw_ads`.
- `--only funnel` - загрузить только воронку в `funnel_analytics`.
- `--only all` - полная синхронизация, используется по умолчанию.

Дополнительные флаги:

- `--skip-ads` - не загружать рекламную статистику.
- `--skip-funnel` - не загружать воронку.

## Как данные преобразуются

### Нормализация строк

Функции `sales_to_sheet_rows`, `orders_to_sheet_rows`, `stocks_to_sheet_rows`, `ads_to_sheet_rows`, `funnel_to_sheet_rows` приводят сырые JSON-ответы API к двумерным таблицам: первая строка содержит заголовки, остальные строки - значения.

Числовые поля приводятся через `Decimal`, чтобы снизить риск ошибок округления в деньгах.

### PnL по дням

Функция `aggregate_daily_pnl` группирует продажи и рекламу по `(date, article)`.

Продажи учитываются со знаком:

- `Продажа` = `+1`;
- `Возврат` = `-1`;
- служебные строки без продаж учитываются только если в них есть хранение, приемка, штрафы, удержания или доплаты.

Основные показатели `daily_pnl`:

- `orders_amount`;
- `sales_amount`;
- `sales_without_spp`;
- `wb_commission`;
- `acquiring_fee`;
- `storage_fee`;
- `acceptance_fee`;
- `penalties`;
- `deductions`;
- `additional_payments`;
- `delivery_fee`;
- `ad_spend`;
- `cogs_amount`;
- `net_profit`;
- `margin_pct`.

Себестоимость берется из SKU и умножается на количество продаж с учетом возвратов.

### Управленческая аналитика

Функция `build_analytics_tables` строит несколько срезов:

- `finance_article_day_detail` - подробная экономика по артикулу и дню.
- `analytics_article_day` - продажи, реклама, прибыль, ДРР и маржа по артикулу и дню.
- `analytics_day` - агрегированная сводка по дням.
- `analytics_article_period` - агрегированная сводка по артикулу за выбранный период.

В расчетах используются продажи по нашей цене, СПП, вознаграждение WB, НДС WB, эквайринг, логистика, хранение, приемка, штрафы, удержания, доплаты, реклама и себестоимость.

### Выкупы по датам заказов

Функция `build_buyout_order_day_rows` связывает:

- `raw_orders` как источник дат заказов и связки `nmId -> article`;
- `raw_sales` как источник продаж, возвратов, дат выкупа и дат исходного заказа;
- `raw_ads` как источник рекламных расходов по дате;
- `funnel_analytics` как более точный источник суммы и количества заказов, выкупов и отмен.

Результат сохраняется в `buyout_order_day`. Таблица помогает отвечать на вопрос: что произошло с товарами, заказанными в конкретную дату, даже если выкуп или возврат произошел позже.

### Планирование

Страница `/analytics/planning` и API `/api/analytics/planning` строят расчетный прогноз до выбранной даты. Отдельная SQLite-таблица для планирования сейчас не создается: таблица на экране собирается на лету из уже сохраненных данных.

Основные источники:

- `SKU` - список товаров, предмет, стратегия, категория, себестоимость, комиссия WB и поле `склад`.
- `raw_stocks` - текущий доступный остаток WB по `quantity`.
- `buyout_order_day` - скорость заказов и сумма заказов за базовый период.
- `funnel_analytics` - WB-выкуп, отмены, выкупленная выручка, ABC-категория по выручке.
- `raw_orders` - цены покупателя/продавца, СПП и история для оценки эластичности.
- `raw_ads` - рекламные расходы для ДРР и рекламной эластичности.
- `funnel_impressions_upload`, если таблица есть, - показы, открытия, корзины и заказы для CTR/CR1/CR2.

Базовые периоды в текущей реализации:

- скорость заказов и средний чек: последние 7 полных дней до последней даты в `buyout_order_day`; самая свежая дата исключается, потому что она может быть неполной;
- `% выкупа WB`: период с 44-го по 14-й день до последней даты, чтобы не брать слишком свежие и еще не дозревшие выкупы/отмены;
- реклама и ДРР: 14 дней, заканчивая за 2 дня до последней даты;
- эластичность цены и рекламы: история `raw_orders`/`raw_ads` до 60 дней.

Расчет по SKU:

- стартовый остаток = сумма `raw_stocks.quantity`;
- складской остаток = поле `склад` из `SKU`, добавляется в доступный остаток через настройку `transit_days`;
- базовая скорость = `Заказы, шт / 7`;
- `% выкупа` = `buyoutCount / (buyoutCount + cancelCount)` из `funnel_analytics`, с fallback на предмет, стратегию, категорию или общий процент;
- средний чек = `Сумма заказов / Заказы, шт`, с fallback на группу;
- изменение цены меняет спрос через рассчитанную эластичность;
- невыкупленные товары возвращаются в продажу через настройку `return_delay_days`;
- прогноз ограничивается доступным остатком с учетом будущих возвратов и складского остатка.

Верхний итоговый блок показывает:

- доступный остаток, себестоимость остатка и стоимость остатка;
- прогноз заказов;
- ожидаемые выкупы;
- ожидаемую выручку;
- недопродажи в штуках и рублях из-за нехватки остатка.

Таблица планирования по SKU содержит, среди прочего:

- SKU/WB-артикул, название, предмет, стратегия, категорию;
- остаток WB, складской остаток, себестоимость и комиссию WB;
- базовую и плановую скорость заказов;
- `% выкупа`, средний чек, текущую цену, СПП и плановую цену;
- эластичность цены и рекламы;
- базовый ДРР, CTR, CR1, CR2;
- потенциальные заказы, прогноз заказов, выкупы, выручку, недопродажи;
- дату исчерпания остатка, если товар заканчивается до конца периода.

API `/api/analytics/planning/export` выгружает текущий расчет в Excel. Сценарии изменения цены, ДРР и конверсий хранятся в браузере пользователя через `localStorage`; в SQLite они не записываются.

### WB-биддер и расписание рекламы

Страница `/ads/bidder` управляет рекламными кампаниями WB по кабинетам. Она не меняет ставки; текущая логика управляет состоянием кампаний: запуск и пауза через WB Advertising API.

Что умеет страница:

- искать артикулы по SKU/nmId и подбирать связанные рекламные кампании;
- показывать метрики кампаний из `raw_ads`: показы, клики, расход, заказы, выручку, CTR, CPC, CPM, CPO и ДРР;
- кэшировать состав кампаний из WB Advertising API;
- задавать для кампании дневной бюджет, автоостановку по расходу и часовой интервал паузы;
- запускать проверку вручную через `/api/ads/executor/run-once`;
- показывать журнал действий и опросов WB.

Исполнитель биддера стартует в `web_app.py` как daemon-thread `wb-ad-bidder`. Он просыпается в начале каждого часа, проходит по всем WB-кабинетам с рекламным токеном и применяет настройки. Если опрос расхода падает, следующий повтор происходит через минуту.

Логика действия:

- если `auto_pause=1`, дневной бюджет больше нуля и `today_spend >= daily_budget`, кампания переводится в `paused`;
- если включено расписание и сейчас активен интервал паузы, кампания переводится в `paused`;
- если расписание включено, интервал паузы не активен и бюджет не превышен, кампания переводится в `active`;
- повторное действие не отправляется, если желаемое состояние уже совпадает с последним сохраненным состоянием.

Расход за день подтягивается из `adv/v3/fullstats` не чаще одного раза в час на кампанию, если не задан ручной `force_spend_sync`. Кампании запускаются через `adv/v0/start`, останавливаются через `adv/v0/pause`; ответ `422` считается `noop`, если WB уже находится в нужном состоянии.

Основные таблицы биддера:

- `ad_bidding_campaign_settings` - настройки кампаний: включенность расписания, бюджет, автостоп, расход за день, интервал паузы.
- `ad_campaign_cache` - кэш названий, статусов, типов и `nmId` кампаний.
- `ad_bidding_actions` - фактические попытки start/pause/noop/error.
- `ad_bidding_state` - последнее желаемое состояние и ошибки по кампании.
- `ad_bidding_log` - человекочитаемый журнал опросов, ошибок и автодействий.
- `ad_bidding_executor_lock` - защита от параллельного запуска исполнителя.
- `ad_bidding_global_settings`, `ad_bidding_rules`, `ad_bidding_rule_campaigns` - глобальное недельное расписание и старый слой правил; часть логики сохранена для совместимости, текущий основной экран работает через настройки кампаний.

API:

- `GET /api/ads/articles?q=...` - поиск артикулов для выбора.
- `GET /api/ads/campaigns?nmid=...&days=7|14|30&refresh=1` - кампании и метрики по товару.
- `GET /api/ads/logs?limit=200` - журнал.
- `GET/POST /api/ads/settings` - глобальное расписание.
- `POST /api/ads/campaign-settings` - настройки конкретной кампании.
- `GET/POST /api/ads/rules`, `POST /api/ads/rules/{id}/toggle` - совместимый API правил.
- `POST /api/ads/executor/run-once` - ручной прогон исполнителя.

### Задачник

Страница `/tasks` - простой канбан внутри web UI. Задачи хранятся в основной SQLite-базе текущего кабинета/платформы и не требуют внешнего сервиса.

Поля задачи:

- `title` - что сделать;
- `why` - зачем это нужно;
- `result` - ожидаемый результат;
- `due_date` - срок;
- `status` - один из `backlog`, `in_progress`, `review`, `done`;
- `created_at`, `updated_at`.

API:

- `GET /api/tasks` - список задач.
- `POST /api/tasks` - создать задачу; обязательны `title`, `why`, `result`, `due_date`.
- `POST /api/tasks/update` - изменить поля или статус.
- `POST /api/tasks/delete` - удалить задачу.

### Контроль цен

Контроль цен состоит из трех частей:

1. `scripts/wb_competitor_finder.py` находит товары-конкуренты для кабинета `hld` через MPSTATS и сохраняет связки в `data/wb_prices.db`, таблицу `competitor_products`.
2. `scripts/wb_price_agent.py` запускается на ноуте и отдает `GET /prices?nm=...` на порту `8100`. Агент сам поднимает Chrome с CDP-портом `9224`, ходит в WB internal API из браузерного контекста и возвращает цену, старую цену, остаток, рейтинг и отзывы.
3. `scripts/wb_price_monitor.py` запускается на сервере, сначала пробует HTTP-агент `http://100.65.13.99:8100`, при недоступности использует CDP-туннель как fallback, и пишет историю в `data/wb_prices.db`.

Таблицы:

- `price_history` - история цен, остатков, рейтингов и отзывов по нашим `nmId` для всех WB-кабинетов.
- `competitor_products` - связка нашего товара и найденных конкурентов.
- `competitor_prices` - история цен конкурентов для кабинета `hld`.

Страница `/analytics/competitor-prices` показывает сводку по нашим товарам и конкурентам: текущую цену, новую расчетную цену, медиану/минимум/максимум конкурентов за последние 7 дней, живые цены через агента и расчет маржинальности с учетом себестоимости, комиссии, логистики, эквайринга, ДРР и прочих затрат.

API:

- `GET /api/competitor-prices/summary?cabinet_id=hld` - сводка по нашим товарам.
- `GET /api/competitor-prices/detail?cabinet_id=hld&nm_id=...` - конкуренты выбранного товара.
- `GET /api/competitor-prices/live-prices?nm=123;456` - живой запрос цен через агента.
- `GET /api/competitor-prices/costs?cabinet_id=hld` - себестоимость и экономические параметры из SKU/воронки.

## SQLite-хранилище

Класс `SQLiteStore` хранит все таблицы в одном файле, по умолчанию `data/wb_sync.db`.

Особенности:

- имена таблиц очищаются от небезопасных символов;
- все колонки создаются как `TEXT`;
- первая строка переданного набора считается заголовком;
- `upsert_table` создает уникальный индекс по ключевым колонкам;
- при `update_existing=True` обновляются только непустые новые значения;
- при `overwrite_existing=True` свежий расчет полностью заменяет старую строку по ключу;
- при `allow_new_columns=True` таблица расширяется новыми колонками;
- `replace_table` полностью удаляет и пересоздает таблицу.

Основные таблицы:

- `SKU`/`sku` - справочник артикулов и себестоимости.
- `raw_sales` - сырой финансовый отчет продаж.
- `raw_orders` - сырой поток заказов.
- `raw_ads` - рекламная статистика.
- `raw_stocks` - текущие остатки.
- `funnel_analytics` - воронка продаж.
- `daily_pnl` - дневной PnL.
- `finance_article_day_detail` - подробная экономика по дню и артикулу.
- `analytics_article_day` - компактная аналитика по дню и артикулу.
- `analytics_day` - дневная сводка.
- `analytics_article_period` - период по артикулам.
- `buyout_order_day` - заказы, выкупы, возвраты и реклама по датам заказов.
- `tasks` - канбан-задачи web UI.
- `ad_bidding_*`, `ad_campaign_cache` - настройки, состояние, действия и логи WB-биддера.

Расчет планирования использует эти таблицы, но не создает постоянную таблицу `planning`: данные страницы и Excel-экспорта формируются динамически в `web_app.py`.

Отдельная база `data/wb_prices.db` хранит контроль цен:

- `price_history` - цены наших товаров.
- `competitor_products` - найденные конкуренты.
- `competitor_prices` - цены конкурентов.

## Ограничения и важные детали

- API Wildberries имеет лимиты. Клиент делает паузы и ретраи при `429`, временных `500/502/503/504`, timeout, SSL и сетевых ошибках.
- Финансовый отчет грузится постранично через `rrdId`.
- Остатки грузятся с курсором по `lastChangeDate`, если ответ достигает лимита.
- Реклама грузится чанками до 50 кампаний и окнами до 31 дня.
- Воронка грузится окнами по 7 дней и чанками по 20 `nmId`; при некоторых ошибках чанк дробится до отдельных `nmId`.
- Основная база SQLite хранит значения как текст, поэтому числовая обработка выполняется в коде при построении отчетов или через явные `CAST` в SQL web UI.
- `raw_stocks` заменяется полностью, остальные сырые таблицы обычно обновляются через upsert.
- Производные таблицы пересчитываются из накопленных данных SQLite, поэтому повторный запуск за новый период расширяет историю, а запуск за старый период обновляет соответствующие ключи там, где включен overwrite/update.
- Планирование является расчетной витриной: изменение фильтров и сценариев не меняет базу и не влияет на синхронизацию.
- Биддер рекламы выполняет реальные start/pause-запросы в WB API. Перед включением автостопа нужно проверить дневной бюджет, интервал паузы и рекламный токен кабинета.
- Контроль цен зависит от ноутбука с Tailscale, Chrome-профилем WB и живыми cookies. При недоступности агента live-цены и мониторинг будут неполными.
- API-токены не должны попадать в логи и документацию.
